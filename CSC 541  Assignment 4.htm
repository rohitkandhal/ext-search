Introduction

The goals of this assignment is to study different methods of external searching. We will do this by comparing an algorithm designed for in-memory efficiency (BST) with an algorithm specifically designed for external searching (B-Tree).

User Interface

In both parts of this assignment the user will communicate with your program via the keyboard (or via file redirection). The user will ask your program to store and retrieve integer numbers that represent numeric keys. We are simulating the creation of a primary key index stored on disk, so only the integer keys need to be recorded in your index file. Since each record contains a single integer key (plus any necessary links to other records in the file), you will use fixed-length records. In fact, this is normally a requirement for both of the search algorithms you will implement.

Your program must understand and respond to the following three commands:

  • add key
    Adds a new record with integer key key to the index file. For example:

         add 712412913

    adds a new record with a key of 712412913 to the index file.

  • find key
    Finds the record with integer key key in the index file, if it exists. If a matching record is found, the program should print on-screen:

         Record key exists.

    otherwise, the program should print on-screen:

         Record key does not exist.
  • end
    Ends the program.

Your program will be executed from the command line with a single argument, the name of the index file to use to store the keys. For example:

  
bst file.bst

would start the BST program and store keys in the index file named file.bst. Similarly,

  
btree file.btree

would start the B-Tree program and store keys in the index file named file.btree.

Note: Normally we would support deletion of records and use a simple availability list to track "holes" in the file. You are not being asked to implement such a feature for this assignment. Consequently, the first thing your program should do is delete any existing index file before it begins. You can do this with the following command:

  
#include <stdlib.h>

system( "del file.bst" );

where file.bst is the name of the file in the current working directory you want to remove.

Binary Search Tree (BST)

In the first half of the assignment you will implement a simple file-based BST to store and retrieve keys. The root node of the BST will be stored at offset 0 (at the front) of the index file. All other nodes (both internal and leaf) will be appended to the index file as they are added by the user. Each node will have the format:

  
struct bst_node {
  int key;     // Key value
  long l;      // File offset of left child node
  long r;      // File offset of right child node
}

l and r are the offset locations from the beginning of the file to the left and right child nodes. This is analogous to the use of left and right pointers for an in-memory BST. If either child doesn't exists, the node's l (or r) field should be set to -1.

Consider the BST shown below. The table to its right represents the index file that represents the BST.

     offset key l r
0 25 12 24
12 15 -1 -1
24 32 -1 -1

If we added 18 to the BST, we would get a new tree and corresponding index file that looked like:

     offset key l r
0 25 12 24
12 15 -1 36
24 32 -1 -1
36 18 -1 -1

We added the new node by:

  • finding the new node's parent in the index file
  • appending the new node to the end of the index file
  • updating the parent's child pointer to reference the new node

To find a node in the BST via the index file, simply start at the root (offset 0), then work through the tree until the target key is found, or an empty subtree is encountered.

Implementation

Implement a program called bst that allows the addition and retrieval of integer keys as described above. Store the structure of the BST in a corresponding disk file. No part of the BST except for the node currently being examined is to be held in-memory; all operations must seek through the index file as necessary to perform their tasks.

As your program runs, time how long it takes to perform each find command. Print both the total time spent executing finds and the average time per find when the program ends.

In addition to the standard add, find, and end commands, implement a fourth command called print; print will print the current state of the BST as a collection of key/offset pairs. All the nodes on each level of the BST will be printed on a separate line. Each line will be preceded by its level (where the root node is at level 1), followed by a colon, followed by the key and offset of each node stored at that level in the BST from left to right. For example, the second BST shown above would print as:

  
1: 25/0
2: 15/12 32/24
3: 18/36

where the first value before the slash represents the node's key, and the second value after the slash represent the node's offset.

Hint: If you're wondering how to implement the print command, don't use recursion. Use a FIFO queue that initially holds only the root node. Remove the node at the head of the queue and add that node's left and right children (if they exist) to the end of the queue. Print the node's key and offset, then discard the node. Continue until the queue is empty.

B-Tree

In the second half of the assignment you will implement an order-33 B-Tree (that is, a B-Tree with nodes that hold up to 32 keys and 33 child pointers). The B-Tree will respond to exactly the same commands as the BST described in Part 3 of the assignment. Initially, the root node of the B-Tree will be stored at offset 0 (at the front) of the index file. However, as node splits occur new root nodes may be created and appended to the end of the index file. Therefore, your program will most likely need to use an internal variable to maintain the offset of the current root node.

As an example, a node in an order-33 B-Tree would have the following format:

  
struct btree_node {
  int n;                // Number of keys in node
  int key[ 32 ];        // Key values
  long child[ 33 ];     // File offsets of child nodes
}

child contains the offset locations from the beginning of the file to the (up to) 33 child nodes. This is analogous to the use of l and r offsets for a BST. If any child doesn't exists, its child entry should be set to -1.

Consider the order-4 B-Tree shown below. The table below it represents the index file that represents the B-Tree. For simplicity, we assume blank child entries in the table represent offsets of -1 (no child node).

offset n child[0] key[0] child[1] key[1] child[2] key[2] child[3]
0 1 32 M 64        
32 2 96 D 128 H 160    
64 3 192 Q 224 T 256 X 298
96 2   B   C      
128 2   F   G      
160 3   J   K   L  
192 2   N   P      
224 2   R   S      
256 2   V   W      
298 2   Y   Z      

Implementation

Implement a program called btree that allows the addition and retrieval of integer keys in an order-33 B-Tree as described above. Store the structure of the B-Tree in a corresponding disk file. No part of the B-Tree except for the node currently being examined is to be held in-memory; all operations must seek through the index file as necessary to perform their tasks.

As with the BST, time how long it takes to perform each find as your program runs. Print both the total time spent executing finds and the average time per find when the program ends.

In addition to the standard add, find, and end commands, implement a fourth command called print; print will print the current state of the B-Tree as a collection of key,key,key,key/offset pairs. All the nodes on each level of the B-Tree will be printed on a separate line. Each line will be preceded by its level (where the root node is at level 1), followed by a colon, followed by the keys and offset of each node stored at that level in the B-Tree from left to right. For example, the B-Tree shown above would print as:

  
1: M/0
2: D,H/32 Q,T,X/64
3: B,C/96 F,G/128 J,K,L/160 N,P/192 R,S/224 V,W/256 Y,Z/298

where the values before the slash represents the node's keys, and the value after the slash represent the node's offset. As with the BST, you can use a FIFO queue to simplify implementation of the print command.
